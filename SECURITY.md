# Security Policy

## Supported Versions

The following versions of KONGALI SECURITY are currently supported with security updates:

| Version | Supported |
|---|---|
| `0.1.x` | ✅ Yes |
| `< 0.1.0` | ❌ No |

Security support for future versions will be updated as the project evolves.

---

## Reporting a Vulnerability

If you discover a security vulnerability in KONGALI SECURITY, please **do not disclose it publicly through GitHub Issues, Pull Requests, or public discussions**.

We strongly encourage responsible disclosure.

Please report security vulnerabilities privately to the project maintainers.

### Please include the following information

- A clear description of the vulnerability
- The affected component or module
- The affected version
- Steps to reproduce the issue
- Proof of concept, if applicable
- Potential security impact
- Suggested mitigation or remediation, if available

Please avoid including sensitive personal information or real-world credentials in vulnerability reports.

---

## Responsible Disclosure

We ask security researchers and contributors to:

1. Give the maintainers reasonable time to investigate the issue.
2. Avoid accessing, modifying, deleting, or exposing data that does not belong to you.
3. Avoid disrupting production systems or third-party infrastructure.
4. Avoid testing against systems without explicit authorization.
5. Avoid publicly disclosing the vulnerability before a fix or mitigation is available.

We appreciate responsible security research and coordinated disclosure.

---

## Security Response Process

When a vulnerability report is received, maintainers will:

1. Acknowledge receipt of the report.
2. Review and validate the reported vulnerability.
3. Assess the severity and potential impact.
4. Identify affected versions and components.
5. Develop and test a security fix or mitigation.
6. Prepare a security advisory when appropriate.
7. Release the fix through the appropriate project channels.
8. Publicly disclose relevant information after remediation, when appropriate.

The exact response timeline may depend on the severity and complexity of the vulnerability.

---

## Security Severity

Security issues may be evaluated using the following general severity levels:

| Severity | Description |
|---|---|
| Critical | Vulnerabilities that may allow severe compromise, remote code execution, or major security impact |
| High | Vulnerabilities that may significantly compromise confidentiality, integrity, or availability |
| Medium | Vulnerabilities with meaningful but limited security impact |
| Low | Vulnerabilities with limited security implications |
| Informational | Security improvements or hardening recommendations |

Where appropriate, the maintainers may use the Common Vulnerability Scoring System (CVSS) to help assess severity.

## Scope

The security policy applies primarily to:

- KONGALI SECURITY source code
- Official KONGALI SECURITY releases
- Official project dependencies and configurations
- Security-related workflows and automation
- Official documentation that may affect secure deployment or usage

Third-party systems, external infrastructure, and unrelated repositories are outside the scope of this policy unless explicitly stated otherwise.

## Authorized Use

KONGALI SECURITY is intended for:

- Authorized security testing
- Defensive cybersecurity operations
- Security research
- Threat intelligence analysis
- Security automation
- Digital forensics and analysis
- Education and training
- System administration
- Incident response

Users are responsible for ensuring that their activities comply with all applicable laws, regulations, organizational policies, and authorization requirements.

**Do not use KONGALI SECURITY to access, scan, monitor, attack, or interfere with systems without proper authorization.**

---

## Security Best Practices

Users are encouraged to:

- Keep KONGALI SECURITY updated.
- Use trusted and verified dependencies.
- Protect API keys and credentials.
- Never commit secrets to the repository.
- Use environment variables for sensitive configuration.
- Restrict access to security-sensitive systems.
- Review generated reports before sharing them.
- Avoid exposing sensitive logs or threat intelligence data publicly.
